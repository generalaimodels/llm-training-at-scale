Pipeline Parallelism: Comprehensive Technical Exposition

1. Motivation: Why Pipeline Parallelism?

1.1 The Inter-Node Communication Bottleneck

Tensor Parallelism (TP) partitions individual layer weight matrices across GPUs within a single node. However, scaling TP beyond the GPUs on a single node (typically Nintra=4N_{\text{intra}} = 4Nintra​=4or888) forces communication across inter-node network links. These links operate at significantly lower bandwidth than intra-node interconnects (e.g., NVLink at $\sim$900 GB/s vs. InfiniBand at $\sim$50–400 GB/s), degrading performance during collective operations such as all-reduce, all-gather, and reduce-scatter.

Key empirical observation: When benchmarking all-reduce across multiple nodes (each with 8 GPUs), median bandwidth drops substantially as node count increases, and variance (5th–95th percentile spread) widens—confirming that inter-node communication is a primary scaling bottleneck for TP.

1.2 The Model Size Problem

For large models (≥70B\geq 70\text{B}≥70Bparameters), the memory footprint of weights alone can exceed the aggregate GPU memory of a single node. Specifically, for a model withPPP parameters stored in mixed precision (e.g., 2 bytes per parameter for BF16):

Weight Memory=P×2  bytes \text{Weight Memory} = P \times 2 \;\text{bytes} Weight Memory=P×2bytes

For P=70×109P = 70 \times 10^9P=70×109:

Weight Memory=70×109×2=140  GB \text{Weight Memory} = 70 \times 10^9 \times 2 = 140 \;\text{GB} Weight Memory=70×109×2=140GB

This exceeds the memory capacity of 4–8 GPUs (e.g., 4 × 80 GB = 320 GB leaves little room for optimizer states, gradients, and activations). Sequence parallelism and context parallelism address long-sequence memory pressure but do not address the fundamental weight memory constraint.

Pipeline Parallelism (PP) resolves this by partitioning the model along the depth (layer) dimension across multiple GPUs.

2. Core Concept: Partitioning Layers Across GPUs

2.1 Definition

Pipeline parallelism splits a model’s LLLtransformer layers intopppcontiguous groups called stages, distributing each stage to a separate GPU (or device). If we havepppGPUs andLLL layers:

Layers per stage=Lp \text{Layers per stage} = \frac{L}{p} Layers per stage=pL​

Example: For L=32L = 32L=32layers andp=8p = 8p=8 GPUs:

GPU Layers
GPU 0 Layers 1–4
GPU 1 Layers 5–8
GPU 2 Layers 9–12
GPU 3 Layers 13–16
GPU 4 Layers 17–20
GPU 5 Layers 21–24
GPU 6 Layers 25–28
GPU 7 Layers 29–32

2.2 Memory Reduction for Model Parameters

Each GPU stores only 1p\frac{1}{p}p1​of the model’s parameters. For an 8B parameter model withp=4p = 4p=4:

Parameters per GPU=8×1094=2×109 \text{Parameters per GPU} = \frac{8 \times 10^9}{4} = 2 \times 10^9 Parameters per GPU=48×109​=2×109
Weight memory per GPU=P×2p  bytes \text{Weight memory per GPU} = \frac{P \times 2}{p} \;\text{bytes} Weight memory per GPU=pP×2​bytes

2.3 Activation Memory: No Savings

A critical and initially counterintuitive observation: activation memory is NOT reduced by pipeline parallelism.

Explanation: Each GPU handles 1p\frac{1}{p}p1​of the layers, so the activation memory per micro-batch per stage isactivsp\frac{\text{activs}}{p}pactivs​. However, in any PP schedule, each GPU must perform ppp (or more) forward passes on successive micro-batches before beginning the first backward pass. Therefore, the total activation memory stored simultaneously on each GPU is:

Activation memory per GPU=p×activsp=activs \text{Activation memory per GPU} = p \times \frac{\text{activs}}{p} = \text{activs} Activation memory per GPU=p×pactivs​=activs

where activs\text{activs}activs denotes the total activation memory for a single micro-batch across all layers. The activation memory per GPU thus remains approximately equal to the non-parallelized case.

2.4 Communication Pattern

Unlike data parallelism (which communicates gradients) or ZeRO-3 (which communicates parameters), pipeline parallelism communicates activation tensors sequentially between adjacent stages. Between stage sssand stages+1s+1s+1, the output activation tensor of the last layer on GPU sssis sent as input to the first layer on GPUs+1s+1s+1.

PP Communication=point-to-point send/recv of activation tensors at stage boundaries \text{PP Communication} = \text{point-to-point send/recv of activation tensors at stage boundaries} PP Communication=point-to-point send/recv of activation tensors at stage boundaries

Advantage over TP: Communication occurs only at p−1p - 1p−1stage boundaries (once per stage transition), rather than multiple times within every layer. The volume per communication is moderate (one hidden-state tensor of shape[Bμ,S,H][B_{\mu}, S, H][Bμ​,S,H]whereBμB_{\mu}Bμ​is micro-batch size,SSSis sequence length,HHH is hidden dimension).

3. The Pipeline Bubble: Fundamental Inefficiency

3.1 Naive Scheduling (Single Micro-Batch)

When a single batch is processed through ppp stages sequentially, only one GPU is active at any given time. All other GPUs are idle.

Timing definitions:

  • tft_ftf​: Time for a forward pass through one stage for one micro-batch
  • tbt_btb​: Time for a backward pass through one stage for one micro-batch
  • Common approximation: tb≈2×tft_b \approx 2 \times t_ftb​≈2×tf​ (backward involves computing gradients w.r.t. both inputs and weights)

Ideal time (if perfectly parallelized):

tideal=tf+tb t_{\text{ideal}} = t_f + t_b tideal​=tf​+tb​

Pipeline bubble time (additional idle time):

tbubble=(p−1)×(tf+tb) t_{\text{bubble}} = (p - 1) \times (t_f + t_b) tbubble​=(p−1)×(tf​+tb​)

This represents the cumulative idle time across all stages where GPUs wait while others compute.

Bubble ratio (fraction of wasted time relative to ideal):

rbubble=(p−1)×(tf+tb)tf+tb=p−1 \boxed{r_{\text{bubble}} = \frac{(p - 1) \times (t_f + t_b)}{t_f + t_b} = p - 1} rbubble​=tf​+tb​(p−1)×(tf​+tb​)​=p−1​

For p=8p = 8p=8: rbubble=7r_{\text{bubble}} = 7rbubble​=7, meaning the bubble time is 7×7\times7× the ideal compute time. This is catastrophically inefficient.

4. All Forward, All Backward (AFAB) Schedule

4.1 Concept

To mitigate the bubble, we split the global batch into mmm micro-batches. The schedule proceeds as:

  1. All Forward Phase: Process all mmm micro-batches through the forward pass sequentially across stages.
  2. All Backward Phase: Process all mmm micro-batches through the backward pass sequentially.

When GPU s+1s+1s+1begins processing micro-batchiii, GPU ssscan immediately start processing micro-batchi+1i+1i+1. This creates a pipelined overlap.

4.2 Bubble Analysis

Ideal time for mmm micro-batches:

tideal=m×(tf+tb) t_{\text{ideal}} = m \times (t_f + t_b) tideal​=m×(tf​+tb​)

Bubble time remains:

tbubble=(p−1)×(tf+tb) t_{\text{bubble}} = (p - 1) \times (t_f + t_b) tbubble​=(p−1)×(tf​+tb​)

Bubble ratio:

rbubble=(p−1)×(tf+tb)m×(tf+tb)=p−1m \boxed{r_{\text{bubble}} = \frac{(p - 1) \times (t_f + t_b)}{m \times (t_f + t_b)} = \frac{p - 1}{m}} rbubble​=m×(tf​+tb​)(p−1)×(tf​+tb​)​=mp−1​​

By increasing mmm, the bubble ratio decreases inversely. For p=8p = 8p=8, m=32m = 32m=32:

rbubble=732=0.21875≈21.9% r_{\text{bubble}} = \frac{7}{32} = 0.21875 \approx 21.9\% rbubble​=327​=0.21875≈21.9%

4.3 Memory Problem

AFAB requires storing activations for all mmm micro-batches simultaneously, because no backward pass begins until all forward passes complete. The activation memory requirement on each GPU is:

Activation memoryAFAB=m×activsp \text{Activation memory}_{\text{AFAB}} = m \times \frac{\text{activs}}{p} Activation memoryAFAB​=m×pactivs​

Since we want large mmm to reduce the bubble, this creates a memory explosion—activations for many micro-batches must be retained until the backward phase begins.

5. One Forward, One Backward (1F1B) Schedule

5.1 Concept

The 1F1B schedule addresses the activation memory problem by starting backward passes as early as possible. The schedule has three phases:

  1. Warm-up phase: Each GPU performs successive forward passes to fill the pipeline (the number of warm-up forward passes depends on the stage position).
  2. Steady-state phase: Each GPU alternates between one forward pass and one backward pass (hence “1F1B”).
  3. Cool-down phase: Each GPU drains remaining backward passes.

5.2 Bubble Analysis

The bubble size in 1F1B is identical to AFAB:

rbubble=p−1m \boxed{r_{\text{bubble}} = \frac{p - 1}{m}} rbubble​=mp−1​​

The bubble is not reduced because the total amount of idle time at the start (warm-up) and end (cool-down) remains the same—it is simply rearranged.

5.3 Memory Advantage

The critical improvement is in activation memory. In 1F1B, each GPU stores activations for at most pppmicro-batches (notmmm micro-batches as in AFAB):

Activation memory1F1B=p×activsp=activs \text{Activation memory}_{\text{1F1B}} = p \times \frac{\text{activs}}{p} = \text{activs} Activation memory1F1B​=p×pactivs​=activs

In contrast to AFAB where activation memory was m×activspm \times \frac{\text{activs}}{p}m×pactivs​, 1F1B limits it to p×activspp \times \frac{\text{activs}}{p}p×pactivs​. Since typically m≫pm \gg pm≫p in practical configurations, this is a substantial memory reduction.

Because 1F1B uses less activation memory, we can increase mmmfurther (without running out of memory), which indirectly reduces the bubble ratiop−1m\frac{p-1}{m}mp−1​.

5.4 Practical Scaling Behavior

Empirical benchmarks reveal two regimes:

Configuration Behavior
m≤p−1m \leq p - 1m≤p−1 Bubble dominates; performance degrades asppp increases
m=32≫p−1m = 32 \gg p - 1m=32≫p−1 Performance improves at lowppp; still limited at very large ppp

Cross-node scaling advantage: When scaling from one node (p=8p = 8p=8) to two nodes (p=16p = 16p=16), PP shows only $\sim$14% performance drop, compared to $\sim$43% for TP. This is because PP communicates only point-to-point activation tensors at stage boundaries, whereas TP requires bandwidth-intensive collective operations (all-reduce) within every layer.

5.5 Implementation Complexity

In 1F1B, forward and backward passes are no longer globally sequential. Different GPUs execute forward and backward passes for different micro-batches concurrently. This requires:

  • Per-device scheduling logic to decide when to switch between forward and backward execution.
  • Extensive modifications to both training loop code and model code.
  • Careful management of micro-batch indexing and gradient accumulation.

6. Interleaved Stages

6.1 Concept

Instead of assigning contiguous layer blocks to each GPU, interleaved PP assigns non-contiguous layer subsets. Each GPU hosts vvvmodel chunks (also called virtual stages), wherevvv is the number of chunks per GPU.

Example with L=8L = 8L=8layers,p=2p = 2p=2GPUs,v=2v = 2v=2 chunks per GPU:

GPU Chunks Layers
GPU 0 Chunk 0, Chunk 2 Layers 1–2, Layers 5–6
GPU 1 Chunk 1, Chunk 3 Layers 3–4, Layers 7–8

A micro-batch now loops through the GPUs multiple times during a single forward pass: GPU 0 → GPU 1 → GPU 0 → GPU 1 → …

6.2 Bubble Reduction

Each forward and backward pass through a single chunk is vvv times shorter than a full stage pass. The pipeline bubble time becomes:

tbubble=(p−1)×(tf+tb)v t_{\text{bubble}} = \frac{(p - 1) \times (t_f + t_b)}{v} tbubble​=v(p−1)×(tf​+tb​)​

The bubble ratio:

rbubble=1v⋅(p−1)×(tf+tb)m×(tf+tb)=p−1v⋅m \boxed{r_{\text{bubble}} = \frac{1}{v} \cdot \frac{(p - 1) \times (t_f + t_b)}{m \times (t_f + t_b)} = \frac{p - 1}{v \cdot m}} rbubble​=v1​⋅m×(tf​+tb​)(p−1)×(tf​+tb​)​=v⋅mp−1​​

where:

  • ppp = pipeline parallelism degree (number of GPUs)
  • mmm = number of micro-batches
  • vvv = number of model chunks (virtual stages) per GPU

6.3 Communication Trade-off

The number of point-to-point communications increases by a factor of vvv, since each micro-batch traverses each GPU vvv times instead of once. This introduces a direct trade-off:

Communication volume∝v×(p−1) \text{Communication volume} \propto v \times (p - 1) Communication volume∝v×(p−1)
Bubble size∝p−1v⋅m \text{Bubble size} \propto \frac{p - 1}{v \cdot m} Bubble size∝v⋅mp−1​

The optimal vvv balances reduced idle time against increased communication overhead.

6.4 Scheduling Policies: Depth-First vs. Breadth-First

With interleaved stages, a scheduling decision arises at each time step for each GPU: should it prioritize:

Policy Description Effect
Depth-first Advance earlier micro-batches through later layers first Minimizes per-micro-batch latency; completes individual micro-batches faster, freeing activation memory sooner
Breadth-first Advance later micro-batches through earlier layers first Maximizes pipeline filling; keeps all stages busy

The Llama 3.1 training pipeline uses a 1F1B schedule with interleaved stages, with a tunable priority parameter between depth-first and breadth-first policies.

7. Zero Bubble Pipeline Parallelism

7.1 Key Insight: Decomposing the Backward Pass

The backward pass through a linear layer Y=XW\mathbf{Y} = \mathbf{X}\mathbf{W}Y=XW involves two independent gradient computations:

  1. Input gradient (BBB): Gradient w.r.t. the input activations X\mathbf{X}X, needed to propagate gradients to earlier layers.
∂L∂X=∂L∂Y⋅W⊤ \frac{\partial \mathcal{L}}{\partial \mathbf{X}} = \frac{\partial \mathcal{L}}{\partial \mathbf{Y}} \cdot \mathbf{W}^{\top} ∂X∂L​=∂Y∂L​⋅W⊤
  1. Weight gradient (WWW): Gradient w.r.t. the weight matrix W\mathbf{W}W, needed for the optimizer update.
∂L∂W=X⊤⋅∂L∂Y \frac{\partial \mathcal{L}}{\partial \mathbf{W}} = \mathbf{X}^{\top} \cdot \frac{\partial \mathcal{L}}{\partial \mathbf{Y}} ∂W∂L​=X⊤⋅∂Y∂L​

Critical observation: Operation BBBmust complete before the backward pass of the preceding stage can begin (it is on the critical path). However, operationWWW has no such dependency—it only needs to complete before the optimizer step. Therefore:

W can be scheduled anywhere after the corresponding B of the same stage W \text{ can be scheduled anywhere after the corresponding } B \text{ of the same stage} W can be scheduled anywhere after the corresponding B of the same stage

7.2 Exploiting the Decomposition

By splitting the coarse-grained backward pass into fine-grained BBBandWWWoperations, we gain scheduling flexibility. TheWWW operations can be strategically placed into bubble slots that would otherwise be idle.

Timing decomposition:

tb=tB+tW t_b = t_B + t_W tb​=tB​+tW​

where tBt_BtB​is the time for the input gradient computation andtWt_WtW​ is the time for the weight gradient computation.

7.3 ZB-H1 and ZB-H2 Schedules

The Zero Bubble paper proposes two schedules:

Schedule Description Bubble
ZB-H1 Handcrafted schedule with BBB/WWW decomposition Significantly reduced
ZB-H2 Optimized schedule filling all bubbles with WWW operations Theoretically zero

In ZB-H2, every idle slot is filled with a WWW computation, achieving:

rbubble≈0 \boxed{r_{\text{bubble}} \approx 0} rbubble​≈0​

7.4 Optimal Scheduling via Integer Linear Programming (ILP)

Finding the optimal placement of FFF(forward),BBB(input backward), andWWW(weight backward) operations acrosspppstages andmmm micro-batches is formulated as an Integer Linear Programming problem:

Objective:

min⁡  tbubble \min \; t_{\text{bubble}} mintbubble​

Subject to constraints:

  1. Data dependency: Fis+1F_i^{s+1}Fis+1​cannot start beforeFisF_i^{s}Fis​ completes (forward propagation order).
  2. Data dependency: BisB_i^{s}Bis​cannot start beforeBis+1B_i^{s+1}Bis+1​ completes (backward propagation order).
  3. Ordering: BisB_i^{s}Bis​cannot start beforeFisF_i^{s}Fis​ completes.
  4. Flexible scheduling: WisW_i^{s}Wis​must occur afterBisB_i^{s}Bis​ but can occur at any later time.
  5. Optimizer constraint: All WisW_i^{s}Wis​for all micro-batchesiii must complete before the optimizer step.
  6. Non-overlap: No two operations on the same GPU can overlap in time.

Here, FisF_i^{s}Fis​denotes the forward pass for micro-batchiiiat stagesss, and similarly for BisB_i^{s}Bis​andWisW_i^{s}Wis​.

8. DualPipe (DeepSeek-V3/R1)

8.1 Concept

DualPipe extends the zero-bubble decomposition by introducing two concurrent pipeline streams propagating from both ends of the pipeline dimension simultaneously:

  • Stream 1: Micro-batches flow forward from stage 0 → stage p−1p-1p−1 (left to right)
  • Stream 2: Micro-batches flow forward from stage p−1p-1p−1 → stage 0 (right to left)

These bidirectional streams are interleaved on each GPU, ensuring that when one stream encounters a dependency stall, the other stream can utilize the idle compute cycles.

8.2 Fine-Grained Operation Decomposition

DualPipe further decomposes operations beyond FFF, BBB, WWW to include communication operations (all-to-all for MoE expert routing in DeepSeek-V3). The scheduling interleaves:

  • Forward computation (FFF)
  • Input backward computation (BBB)
  • Weight backward computation (WWW)
  • All-to-all communication (for expert parallelism)

By overlapping communication with computation from the opposing stream, DeepSeek-V3 achieved:

near-zero all-to-all communication overhead \text{near-zero all-to-all communication overhead} near-zero all-to-all communication overhead

8.3 Complexity

The DualPipe schedule is significantly more complex than 1F1B or interleaved schedules. Its design requires:

  1. Precise profiling of individual operation durations (tFt_FtF​, tBt_BtB​, tWt_WtW​, tcommt_{\text{comm}}tcomm​).
  2. Solving an ILP or heuristic optimization problem for operation placement.
  3. Bidirectional pipeline infrastructure with careful synchronization.

11. Practical Design Principles

  1. Choose m≫p−1m \gg p - 1m≫p−1 to minimize the bubble ratio, subject to the constraint that mmm divides the global batch size.

  2. PP excels across nodes because point-to-point activation transfers tolerate lower inter-node bandwidth far better than TP’s collective all-reduce operations ($\sim$14% degradation vs. $\sim$43% for TP when crossing node boundaries).

  3. Interleaving (v>1v > 1v>1) trades communication for compute efficiency. Increase vvvonly when intra-stage communication bandwidth is sufficient to absorb thevvv-fold increase in transfers.

  4. 1F1B is strictly preferred over AFAB when activation memory is the binding constraint, as it reduces peak activation storage from O(m)\mathcal{O}(m)O(m)toO(p)\mathcal{O}(p)O(p) micro-batches.

  5. Zero-bubble methods require fine-grained profiling and ILP-based scheduling, making them implementation-intensive but near-optimal for large-scale deployments (e.g., DeepSeek-V3/R1).

  6. PP does not reduce activation memory per se—it reduces parameter memory. Activation recomputation (gradient checkpointing) remains the primary tool for activation memory reduction and is orthogonal to PP.
