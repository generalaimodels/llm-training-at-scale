Expert Parallelism and 5D Parallelism: A Comprehensive Technical Treatment

1. Expert Parallelism (EP)

1.1 Prerequisite: Mixture of Experts (MoE) Architecture

In a standard Transformer layer, every token passes through a single feedforward network (FFN). In a Mixture of Experts (MoE) layer, the single FFN is replaced by NEN_ENE​parallel expert FFNs{E1,E2,…,ENE}\{E_1, E_2, \dots, E_{N_E}\}{E1​,E2​,…,ENE​​}and a gating (router) networkG(⋅)G(\cdot)G(⋅) that determines which expert(s) each token should be sent to.

Given an input hidden state h∈Rd\mathbf{h} \in \mathbb{R}^{d}h∈Rd for a single token, the MoE layer output is:

MoE(h)=∑i=1NEgi(h)⋅Ei(h) \text{MoE}(\mathbf{h}) = \sum_{i=1}^{N_E} g_i(\mathbf{h}) \cdot E_i(\mathbf{h}) MoE(h)=i=1∑NE​​gi​(h)⋅Ei​(h)

where gi(h)g_i(\mathbf{h})gi​(h)is the gating weight for expertiii, computed by the router:

gi(h)={softmax(TopK(Wgh))iif i∈TopK indices0otherwise g_i(\mathbf{h}) = \begin{cases} \text{softmax}\bigl(\text{TopK}(\mathbf{W}_g \mathbf{h})\bigr)_i & \text{if } i \in \text{TopK indices} \\ 0 & \text{otherwise} \end{cases} gi​(h)={softmax(TopK(Wg​h))i​0​if i∈TopK indicesotherwise​

Here Wg∈RNE×d\mathbf{W}_g \in \mathbb{R}^{N_E \times d}Wg​∈RNE​×dis the learnable router weight matrix, andTopK\text{TopK}TopKselects theKKKexperts with highest router logits. TypicallyK≪NEK \ll N_EK≪NE​(e.g.,K=2K = 2K=2out ofNE=256N_E = 256NE​=256 in DeepSeek-V3), meaning each token activates only a sparse subset of experts.

Key property: Each expert EiE_iEi​ is a standard FFN:

Ei(h)=Wi(2)⋅σ ⁣(Wi(1)h+bi(1))+bi(2) E_i(\mathbf{h}) = W_i^{(2)} \cdot \sigma\!\left(W_i^{(1)} \mathbf{h} + \mathbf{b}_i^{(1)}\right) + \mathbf{b}_i^{(2)} Ei​(h)=Wi(2)​⋅σ(Wi(1)​h+bi(1)​)+bi(2)​

where Wi(1)∈Rdff×dW_i^{(1)} \in \mathbb{R}^{d_{ff} \times d}Wi(1)​∈Rdff​×d, Wi(2)∈Rd×dffW_i^{(2)} \in \mathbb{R}^{d \times d_{ff}}Wi(2)​∈Rd×dff​, and σ(⋅)\sigma(\cdot)σ(⋅) is a nonlinearity (e.g., SiLU, GeLU). Since each expert is fully independent of every other expert, this creates a natural axis for parallelism.

1.2 Definition of Expert Parallelism

Expert Parallelism (EP) distributes the NEN_ENE​experts acrossWEPW_{EP}WEP​ workers (GPUs), where each worker holds a disjoint subset of experts:

Worker w holds experts: {Ei  |  i∈[(w−1)⋅NEWEP+1,    w⋅NEWEP]} \text{Worker } w \text{ holds experts: } \left\{ E_i \;\middle|\; i \in \left[\frac{(w-1) \cdot N_E}{W_{EP}} + 1, \;\; \frac{w \cdot N_E}{W_{EP}}\right] \right\} Worker w holds experts: {Ei​​i∈[WEP​(w−1)⋅NE​​+1,WEP​w⋅NE​​]}

For NEN_ENE​experts distributed acrossWEPW_{EP}WEP​ workers, each worker stores:

NEWEP experts \frac{N_E}{W_{EP}} \text{ experts} WEP​NE​​ experts

Contrast with Tensor Parallelism (TP): In TP, a single weight matrix W∈Rm×nW \in \mathbb{R}^{m \times n}W∈Rm×n is split (column-wise or row-wise) across workers, requiring synchronized partial matrix multiplications followed by collective operations (AllReduce or AllGather). In EP, each expert’s weight matrices are kept intact on a single worker — no matrix splitting is needed. The only communication required is routing tokens to the correct worker.

1.3 Communication Pattern: All-to-All

The fundamental communication primitive in EP is the All-to-All operation, which occurs twice per MoE layer:

Forward Pass

Step 1 — Dispatch (All-to-All): After the router computes gating decisions, each worker sends tokens destined for remote experts to the appropriate worker. Formally, if worker wwwhas a local batch of tokens{h1,…,hB}\{\mathbf{h}_1, \dots, \mathbf{h}_B\}{h1​,…,hB​}and the router assigns tokenhj\mathbf{h}_jhj​to expertEiE_iEi​residing on workerw′w'w′, then hj\mathbf{h}_jhj​must be communicated from workerwwwto workerw′w'w′.

Step 2 — Compute: Each worker processes received tokens through its local experts.

Step 3 — Combine (All-to-All): The expert outputs are sent back to the originating workers, weighted by gating scores, and summed.

The communication volume per MoE layer for a single token routed to KKK experts is:

Vcomm=2⋅K⋅d(dispatch + combine, per token) V_{\text{comm}} = 2 \cdot K \cdot d \quad \text{(dispatch + combine, per token)} Vcomm​=2⋅K⋅d(dispatch + combine, per token)

For a local batch of BBBtokens acrossWEPW_{EP}WEP​ workers:

Vtotal=2⋅B⋅K⋅d⋅(1−1WEP) V_{\text{total}} = 2 \cdot B \cdot K \cdot d \cdot \left(1 - \frac{1}{W_{EP}}\right) Vtotal​=2⋅B⋅K⋅d⋅(1−WEP​1​)

The factor (1−1WEP)\left(1 - \frac{1}{W_{EP}}\right)(1−WEP​1​) accounts for the fact that tokens assigned to local experts do not require inter-worker communication.

1.4 EP Combined with Data Parallelism (DP)

EP alone only parallelizes the MoE layers. All non-MoE components — self-attention, layer normalization, embeddings — remain replicated across all EP workers. This means:

  • Without DP, every worker processes the same input batch through non-MoE layers → redundant computation.
  • With DP, the input batch is sharded across workers, and each worker processes a different micro-batch through non-MoE layers → no redundancy.

Given WWWtotal GPUs,WEPW_{EP}WEP​GPUs for expert parallelism, andWDPW_{DP}WDP​ GPUs for data parallelism:

W=WEP×WDP W = W_{EP} \times W_{DP} W=WEP​×WDP​

The effective batch size per GPU for non-MoE layers becomes:

Blocal=BglobalWDP B_{\text{local}} = \frac{B_{\text{global}}}{W_{DP}} Blocal​=WDP​Bglobal​​

while each GPU holds NEWEP\frac{N_E}{W_{EP}}WEP​NE​​ experts for MoE layers.

Gradient synchronization: After the backward pass, gradients for non-MoE parameters are synchronized via AllReduce across the WDPW_{DP}WDP​ data-parallel group, while expert gradients are not synchronized across the EP group (since each expert exists on exactly one worker).

DeepSeek-V3 node-bounded routing constraint: To minimize inter-node communication, the router enforces that each token is sent to at most MMMnodes (DeepSeek-V3 usesM=4M = 4M=4). Formally, define the set of nodes as {N1,…,NNnodes}\{\mathcal{N}_1, \dots, \mathcal{N}_{N_{\text{nodes}}}\}{N1​,…,NNnodes​​}, where each node contains a subset of experts. The router first selects the top-MMM nodes by aggregate affinity:

Then the top-KKKexperts are selected only from experts withinMselected\mathcal{M}_{\text{selected}}Mselected​. This ensures that the All-to-All communication is bounded to at most MMM nodes, dramatically reducing cross-node bandwidth requirements:

3.3 Combinability

  • ZeRO-3 + PP: Possible but rarely practical. Combining them requires inflating the global batch size to amortize both weight-transfer overhead (ZeRO-3) and bubble overhead (PP). If combined, ZeRO-3 should cache (keep in memory) the gathered parameters across all PP micro-batches to avoid re-gathering per micro-batch:
Redundant gathers avoided=(nμ−1)×VZeRO-3total \text{Redundant gathers avoided} = (n_{\mu} - 1) \times V_{\text{ZeRO-3}}^{\text{total}} Redundant gathers avoided=(nμ​−1)×VZeRO-3total​
  • ZeRO-1/ZeRO-2 + PP: Naturally complementary. ZeRO-1/2 shard optimizer states and gradients (which are only needed at the update step), while PP shards model layers. No conflicting communication patterns. DeepSeek-V3 uses this combination (PP + ZeRO-1).

4. Interactions Between Parallelism Strategies

4.1 Tensor Parallelism + Sequence Parallelism (TP+SP)

TP shards weight matrices along the hidden dimension ddd:

W=[W(1)∣W(2)∣⋯∣W(WTP)],W(k)∈Rm×(n/WTP) W = [W^{(1)} | W^{(2)} | \cdots | W^{(W_{TP})}], \quad W^{(k)} \in \mathbb{R}^{m \times (n / W_{TP})} W=[W(1)∣W(2)∣⋯∣W(WTP​)],W(k)∈Rm×(n/WTP​)

The matrix multiplication y=Wx\mathbf{y} = W\mathbf{x}y=Wx is computed as:

y=∑k=1WTPW(k)x(k)(row-parallel) \mathbf{y} = \sum_{k=1}^{W_{TP}} W^{(k)} \mathbf{x}^{(k)} \quad \text{(row-parallel)} y=k=1∑WTP​​W(k)x(k)(row-parallel)

or

y(k)=W(k)x(column-parallel) \mathbf{y}^{(k)} = W^{(k)} \mathbf{x} \quad \text{(column-parallel)} y(k)=W(k)x(column-parallel)

requiring AllReduce or AllGather/ReduceScatter operations.

SP complements TP by sharding activations along the sequence dimension LLL in regions where TP is not active (e.g., LayerNorm, Dropout):

X∈RL×d→X(k)∈R(L/WTP)×d,k=1,…,WTP \mathbf{X} \in \mathbb{R}^{L \times d} \rightarrow \mathbf{X}^{(k)} \in \mathbb{R}^{(L/W_{TP}) \times d}, \quad k = 1, \dots, W_{TP} X∈RL×d→X(k)∈R(L/WTP​)×d,k=1,…,WTP​

Why TP should be intra-node: TP operations (AllReduce, AllGather) lie on the critical computation path — the forward pass cannot proceed until the collective is complete. Thus TP demands the highest bandwidth interconnect (NVLink/NVSwitch at 900 GB/s), making it suitable only for intra-node communication:

TTP-comm=2⋅(WTP−1)WTP⋅∣a∣BWintra-node T_{\text{TP-comm}} = \frac{2 \cdot (W_{TP} - 1)}{W_{TP}} \cdot \frac{|\mathbf{a}|}{\text{BW}_{\text{intra-node}}} TTP-comm​=WTP​2⋅(WTP​−1)​⋅BWintra-node​∣a∣​

where ∣a∣|\mathbf{a}|∣a∣ is the activation size per collective operation.

4.3 Expert Parallelism (EP) — Complementary to TP

EP and TP both operate at the sub-layer level but on different sub-layers:

  • TP: Shards the weight matrices of attention projections and (non-expert) FFNs.
  • EP: Shards expert FFNs across workers.

These are naturally non-overlapping and can be combined without conflict. In an MoE Transformer layer:

Self-Attention⏟TP sharded  →  MoE FFN⏟EP sharded  →  LayerNorm \underbrace{\text{Self-Attention}}_{\text{TP sharded}} \;\rightarrow\; \underbrace{\text{MoE FFN}}_{\text{EP sharded}} \;\rightarrow\; \text{LayerNorm} TP shardedSelf-Attention​​→EP shardedMoE FFN​​→LayerNorm

4.4 Similarity Between EP and DP

Both EP and DP involve multiple workers processing different data through different (or identical) parameters:

DP EP
Each worker processes Different micro-batch Different tokens (routed by expert assignment)
Model weights per worker Identical full copy Unique subset of experts
Gradient sync AllReduce across DP group None across EP group (each expert is unique)

Some frameworks treat EP as a specialized form of DP where, instead of identical model replicas processing different batches, each replica holds different experts and receives dynamically routed tokens.

8.2 Combinability Rules (Decision Heuristics)

  1. TP is always intra-node. Assign WTP≤W_{TP} \leqWTP​≤ GPUs per node (typically 4 or 8).

  2. PP spans inter-node with low-bandwidth requirements (only activation tensors at stage boundaries).

  3. ZeRO-1/2 + PP is straightforward and commonly used.

  4. ZeRO-3 + PP is rare; if used, cache gathered weights across micro-batches.

  5. CP is deployed when LLL is very large; it is orthogonal to PP and TP.

  6. EP is deployed when the model has MoE layers; it is orthogonal to TP (different sub-layers).

  7. DP fills the remaining GPU dimensions after all other strategies are assigned:

WDP=WtotalWTP×WPP×WCP×WEP W_{DP} = \frac{W_{\text{total}}}{W_{TP} \times W_{PP} \times W_{CP} \times W_{EP}} WDP​=WTP​×WPP​×WCP​×WEP​Wtotal​​

The global batch size then becomes:

Bglobal=WDP×Bμ×nμ B_{\text{global}} = W_{DP} \times B_\mu \times n_\mu Bglobal​=WDP​×Bμ​×nμ​

which must remain within the range that preserves training convergence (typically validated empirically).
